## 종합 의견

🟡 **조건부 승인** — 방향성과 범위는 적절하나, 13게임 공통 적용을 보장할 명세 정밀도가 부족하다. 아래 🔴 블로커 7건 해소 후 STEP 3 (엔진/공통 위젯 개발) 진입을 권고한다.

핵심 우려:
1. **"진행률" 정의가 게임별로 의미가 다르다** (W1-5) — 사용자 멘탈 모델 충돌
2. **사운드/햅틱/시각 효과 동시 발사** 시점이 명세에 없다 — 산만함 위험
3. **모션 감소(MediaQuery.disableAnimations) 시스템 연동 누락** — a11y 블로커
4. **Mixkit 라이센스 조건이 부정확하게 인용**되었다 (출처 표기 불필요 X) — 법적 리스크

---

## 강점 (유지)

- Wave 1 (8건) + W2-1 (사운드) 묶음 범위가 1주 스코프로 적정
- 공통 위젯 (`AnimatedCell`, `ProgressBar`, `CountUpText`, `AnimatedTrophy`, `SoundManager`) 추상화로 13게임 일관성 확보 의도 명확
- "효과 줄이기" 토글 도입으로 저성능 폰 대응 의지
- audioplayers MIT 선정과 SoundManager의 fail-silent 패턴 (catch 후 무시) 적절
- 기존 SharedPreferences `soundEnabled` 재활용 — 백워드 호환

## 🔴 필수 수정 사항 (블로커)

| # | 항목 | 사유 | 권고 |
|---|------|------|------|
| B1 | **W1-5 진행률 정의 불일치** | "filled/total" vs "lit/whites" vs "revealed/safe" — 의미가 게임마다 다르면 사용자가 "100% 즉시 = 클리어"라는 일관 기대를 가질 수 없다. 특히 노노그램 "채움/정답 채움"은 오답 채움도 카운트되는지 불명. | **"정답에 가까워지는 정도"** 단일 정의로 통일. 각 게임별 정확한 식을 표로 명세하고, 잘못된 입력은 진행률에 포함하지 않는다는 원칙 명문화. 음양/비나이로는 "유효 셀 / 총 셀", 노노그램은 "정답 일치 셀 / 정답 셀". 지뢰찾기는 진행률 바 자체를 비표시(혹은 "안전 셀 공개율") 권장 — 게임 특성상 진척 표시가 스포일러가 될 수 있다. |
| B2 | **모션 감소 시스템 설정 미연동** | `MediaQuery.disableAnimations` 또는 `AccessibilityFeatures.disableAnimations`를 읽지 않으면 a11y 정책 위반. "효과 줄이기"는 앱 내 토글일 뿐. | `SettingsService.effectsReduced || MediaQuery.disableAnimations` 합집합으로 판정. W1-1/W1-2/W1-3/W1-4/W1-7 전부 이 플래그로 단축/생략 분기. 사양에 명시. |
| B3 | **Mixkit 라이센스 조건 오기재** | "출처 표기 불필요"는 부정확. Mixkit Sound Effects License는 **상업 사용 무료**이나, **재배포·재판매 금지** 및 **앱 내부에 한정**된 사용만 허용. CC0가 아님. | 사양에서 "CC0/Mixkit License" 묶음 표현 삭제. 각 효과음별로 (a) 다운로드 URL (b) 라이센스 명 (c) 다운로드 날짜를 `assets/sounds/LICENSES.md`에 기록. CC0만 쓸 거면 Freesound CC0만 사용. Mixkit 사용 시 라이센스 문구 전문 보관. |
| B4 | **사운드/햅틱/시각 효과 동시 발사 시 우선순위 미정의** | 셀 입력 시 (a) 스케일 애니 (b) click 사운드 (c) 같은 숫자 펄스 (d) 행 완성 시 라인 펄스 + 사운드 (e) 마지막 셀 = 게임 완성 사운드 — 동시 5개 효과 가능. 산만 + 사운드 오버랩. | **"한 입력당 사운드 1종"** 규칙 명시: 우선순위 `game_complete > line_complete > mistake > click`. 상위 사운드 재생 시 `click` 스킵. 햅틱도 동일 규칙. 시각 효과는 중첩 허용하되 alpha 0.25 이하로 제한. |
| B5 | **W1-4 shake와 기존 heavyImpact 햅틱 중복 시점 모호** | "햅틱: 기존 heavyImpact 유지"라고만 적혀 시각 shake와 햅틱이 동시인지 순차인지 불명. 또한 셀이 좁은 화면(예: 5인치 폰 노노그램 15×15)에서 ±4px shake는 인접 셀과 겹친다. | shake와 heavyImpact 동시 시작 (t=0). shake 진폭은 **`min(4px, cellSize * 0.08)`** 로 셀 크기 비례. 200ms 내 3회는 frequency 15Hz → 시각적으로 너무 빠름, **2회 / 180ms** 권장 (좌→우→좌→중앙). |
| B6 | **W1-7 배지 팝업이 결과화면 카운트업(W1-6)과 동시 트리거** | 결과 화면 진입 시 카운트업 800ms + 배지 팝업 500ms + badge.wav가 동시에 시작되면 카운트업 사운드 부재로 비대칭, 배지 사운드는 카운트업 중 시작 → 인지 분산. | 시퀀스 정의: t=0 결과 화면 페이드 인 (200ms) → t=200 카운트업 시작 (800ms, 무음) → t=1000 game_complete 사운드 → t=1200 배지 팝업 (있다면) + badge 사운드. 사양에 시퀀스 다이어그램 추가. |
| B7 | **사운드 6종 동시 재생 충돌 정책 없음** | `AudioPlayer` 인스턴스 6개 보유 구조이나, 같은 키 연속 호출 시 `stop()` → `resume()`은 짧은 컷 발생. 다른 키 동시 호출은 OS 믹서 의존 — 안드로이드 일부 기기서 무시됨. | `play(key)`에서 (a) 같은 키 재호출은 200ms throttle (b) `low_latency` 모드 사용 (`AudioPlayer().setPlayerMode(PlayerMode.lowLatency)`) (c) 6개 인스턴스 사전 생성 + `setReleaseMode(ReleaseMode.stop)` 명시. SoundManager 코드에 반영. |

| # | 항목 | 사유 | 권고 |
|---|------|------|------|
| R1 | W1-1 `Curves.easeOutBack` 오버슈트 1.15는 셀 안 숫자 가독성 저하 | easeOutBack은 약 1.7배 오버슈트하므로 1.0→1.15→1.0 곡선과 충돌. 실제 피크가 ~1.25 됨. | `Curves.easeOutCubic` + 수동 Tween(1.0→1.12→1.0) 또는 `TweenSequence` 사용. 사양에 명시. |
| R2 | W1-2 alpha 0.25 초록 펄스가 다크모드에서 흐릿 | 라이트/다크 모드 동일 alpha는 다크에서 약함. | 라이트 0.25 / 다크 0.35 분기. `AppColors.successPulse` 토큰 신설. |
| R3 | W1-3 "같은 숫자 보드 펄스" 100ms는 너무 짧아 깜빡임 인식 안 됨 | 인간 시각 인지 임계 ~150ms. | 150ms로 상향, fade-in 50ms / hold 50ms / fade-out 50ms. |
| R4 | W1-5 LinearProgressIndicator 4px 높이 + Material 토큰만으로 충분치 않음 | 진행률 색상이 게임별 테마와 충돌할 수 있음. 또 100% 도달 시 마지막 펄스 부재. | 진행률 100% 도달 시 200ms 글로우(alpha 0.3 → 0). 색상은 각 게임의 primaryColor 사용. |
| R5 | W1-6 카운트업 시간 단위 처리 미정의 | "5분 32초"를 카운트업 시 332초로 셀지, mm:ss 포맷 유지하며 증가시킬지 불명. | mm:ss는 ss만 카운트업, mm은 ss=60 도달 시 +1. 또는 전체를 ms 단위 정수 카운트 후 포맷 변환. |
| R6 | W1-7 회전 -10° → 0°는 좌우 비대칭 인지 | 사용자가 어색하게 느낄 수 있음. | -10° → +3° → 0° 오버슈트 (스프링 느낌) 또는 회전 제거하고 글로우 강화. |
| R7 | W1-8 페이드 0.3 opacity 비활성화 셀 — 탭 가능 여부 불명 | "햅틱 없음"만 있고 탭 자체 차단 여부 미정. | 탭 자체는 허용(실수 입력 보호용)하되 `onTap` 무시 + 햅틱 없음. 시각만 비활성. |
| R8 | 일시정지/재개 효과음 누락 | 사용자 컨트롤 피드백으로 권장. | `pause.wav` (200ms 하강음) / `resume.wav` (200ms 상승음) 2종 추가 — 총 8종. 또는 동일 click 재사용으로 최소화. |
| R9 | 사운드 토글과 햅틱 토글의 위치/순서 명세 부재 | 설정 화면 IA가 사양에 없음. | 설정 화면 "피드백" 섹션 신설: 사운드 / 햅틱 / 효과 줄이기 3개를 그룹화. 토글 순서 명시. |
| R10 | 첫 사운드 재생 지연 (cold start) 명세 부재 | preload 시점이 main.dart에서 비동기라면 첫 셀 입력이 preload보다 빠를 경우 click 사운드 누락. | preload 완료 전 `play()` 호출은 큐잉하지 말고 무시(fail-silent). 단, preload는 splash 동안 await 권장. |

## 답변 필요한 질문

1. **Mixkit 라이센스 재확인**: 실제 다운로드한 효과음 6종의 라이센스 페이지 URL을 GD가 확인했는가? Mixkit Sound Effects License 전문 첨부 가능한가?
2. **진행률 바 표시 정책**: 지뢰찾기에서 진행률 표시가 스포일러(난이도 체감 저하)가 되지 않는지 GC 검토 필요. 표시 / 비표시 / 선택형 중 선택?
3. **모션 감소 정책**: 시스템 모션 감소 ON 시 사운드도 함께 줄일 것인가? (일반적으로는 별개)
4. **사운드 ON 기본값 적정성**: 기존 사용자(R0~R1 플레이어)에게 v1.0.1+6 업데이트 후 갑자기 사운드가 켜진 상태가 적절한가? **첫 실행 시 "사운드를 켤까요?" 1회 다이얼로그** 또는 **기본 OFF + 설정에서 ON** 중 GC와 합의 필요.
5. **APK 크기 영향**: 600KB 추정의 근거 (각 100KB)는 압축 후인가 원본인가? Play Store는 압축 후 기준.
6. **audioplayers 6.0과 minSdk 31 실기기 테스트 결과**: 이미 진행된 PoC가 있는가? 없다면 W2-1 진입 전 1일 spike 권장.
7. **W1-7 "배지 글로우"의 구체 구현**: BoxShadow blur 값, 색상, 지속시간 미명세.
