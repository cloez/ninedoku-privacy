# Ninedoku 12개 퍼즐 확장 마스터 플랜

> 최종 합의: 2026-06-01
> 버전: 1.0
> PM 에이전트 관리 문서

---

## 1. 프로젝트 개요

### 1.1 목표
Ninedoku를 단일 스도쿠 앱에서 **13개 논리 퍼즐을 포함하는 오프라인 퍼즐 플랫폼**으로 확장한다.

### 1.2 핵심 제약
- 인터넷 권한 없음 (완전 오프라인)
- Firebase/광고/분석 SDK 없음
- 개인정보 수집 없음
- Android 12+ (minSdk 31), iOS 동시 빌드
- Flutter + Dart, Riverpod, go_router

### 1.3 현재 상태 (v1.0.0+3)
- 스도쿠: 5개 모드, 6단계 난이도, 14개 배지, 4개 국어, 640개 테스트 통과
- GitHub: https://github.com/cloez/ninedoku-privacy/sudoku/
- Play Store: 비공개 테스트 (Alpha) 진행 중

- 같은 인터랙션 유형의 게임을 연달아 출시하지 않는다
- 매 릴리스마다 사용자에게 "다른 느낌"을 제공한다
- 예외: R1(Binairo)과 R3(Yin-Yang)은 엔진 공유로 2개 사이에 R2를 끼움

### 4.2 공유 엔진 인터페이스

```dart
/// 모든 퍼즐 게임이 구현하는 추상 인터페이스
abstract class PuzzleEngine<B, S> {
  B generatePuzzle({required int difficulty, required int seed});
  bool validate(B board);
  S solve(B board);
  bool hasUniqueSolution(B board);
  HintResult getHint(B board, S solution, {int level = 1});
  int evaluateDifficulty(B board);
}
```

## 5. 시스템 설계

### 5.1 공유 vs 독립

| 시스템 | 범위 | 설명 |
|--------|------|------|
| 테마 (6개) | 공유 | 앱 전체 일관된 시각 경험 |
| 설정 (전역) | 공유 | 사운드, 진동, 글자 크기, 언어 |
| 설정 (게임별) | 독립 | 스도쿠 '실수 표시', 노노그램 '자동 X' 등 |
| 통계 | 독립 | 게임별 완료 수, 평균 시간, 기록 |
| 배지 (게임별) | 독립 | 게임별 8~14개 |
| 배지 (크로스) | 공유 | 크로스 게임 배지 10개 |
| 오늘의 퍼즐 | 독립 | 게임별 일일 퍼즐 (시드 기반) |
| 타이머/등급 | 공유 | S/A/B/C 등급 체계 |
| 백업/복원 | 공유 | 하나의 JSON에 전체 데이터 포함 |
| 다국어 | 공유 | 통합 문자열 관리 (4개국어) |
| 난이도 | 독립 | 게임별 스케일이 다름 |

### 5.2 크로스 게임 배지 (10개)

| 배지 | 아이콘 | 조건 |
|------|--------|------|
| 호기심 | 🌱 | 3가지 퍼즐 각 1게임 완료 |
| 다재다능 | 🌿 | 5가지 퍼즐 각 1게임 완료 |
| 만능 퍼즐러 | 🌳 | 전체 퍼즐 각 1게임 완료 |
| 하루의 달인 | ⚡ | 하루에 3가지 오늘의 퍼즐 완료 |
| 퍼즐 마라톤 | 🔥 | 하루에 전체 오늘의 퍼즐 완료 |
| 퍼펙트 콜렉터 | 💎 | 5가지 퍼즐에서 S등급 |
| 마스터 오브 마스터 | 🏆 | 3가지 퍼즐의 최고 난이도 클리어 |
| 7일 다양성 | 📅 | 7일 연속 매일 다른 퍼즐 플레이 |
| 100게임 클럽 | 🎯 | 전체 합산 100게임 완료 |
| 1000게임 클럽 | 👑 | 전체 합산 1000게임 완료 |

### 5.3 인터랙션 패턴 (3가지)

#### 패턴 A: 이진 토글
```
빈 칸 → 탭 → 상태1 → 탭 → 상태2 → 탭 → 빈 칸
```
적용: Binairo(0/1), Yin-Yang(흑/백), Minesweeper(깃발/열기), Tents(텐트/잔디), Star Battle(별/X), Light Up(전구/X), Nonograms(채움/X)

#### 패턴 B: 숫자 입력
```
셀 탭 → 하단 숫자 패드에서 숫자 선택
```
적용: Sudoku, Killer Sudoku, Jigsaw Sudoku, Futoshiki, Skyscrapers, Kakuro

#### 패턴 C: 노노그램 전용
```
셀 탭(채움) + 길게 누르기(X) + 드래그(연속 채움)
```
적용: Nonograms

## 7. 개발 프로세스 — 에이전트 리뷰 체계

### 7.1 에이전트 역할 정의

| 에이전트 | 약칭 | 역할 | 품질 게이트 |
|----------|------|------|------------|
| **PM** | PM | 전체 프로세스 통제, 단계 간 전환 승인, 마스터 플랜 준수 감시 | 모든 단계의 진입/탈출 승인 |
| **게임 전문 기획자** | GD | 게임 규칙 정의, 난이도 설계, 배지/모드 기획, 재미 요소 검증 | 기획서 승인 없이 개발 착수 불가 |
| **UX 기획자** | UX | 화면 흐름, 인터랙션 설계, 터치 패턴, 접근성, 다국어 UI 검토 | UX 명세 승인 없이 UI 개발 불가 |
| **개발자** | DEV | 엔진 구현, UI 구현, 테스트 작성, 빌드 | 코드 작성 및 테스트 통과 |
| **전문 테스터** | QA | 기능 테스트, 리그레션 테스트, 엣지 케이스, 오프라인/백키 검증 | 테스트 불합격 시 DEV로 반려 |
| **게임 챔피언** | GC | 실제 플레이어 관점에서 재미/밸런스/UX 최종 검증, 난이도 적절성 확인 | 최종 승인 없이 빌드/배포 불가 |

### 7.3 반려(Reject) 프로세스

반려 발생 시, 해당 단계로 돌아가 수정 후 후속 단계를 **다시** 거쳐야 한다.

```
반려 흐름 예시:

GC가 "힌트가 도움이 안 된다" 판정 (STEP 6 → STEP 1 반려)
  → GD: 힌트 설계 수정
  → UX: 힌트 표시 방식 수정 (필요 시)
  → DEV: HintEngine 수정 + 테스트
  → QA: 변경 부분 재테스트
  → GC: 재검수
  → PASS → STEP 7 진행

QA가 "리그레션 실패" 판정 (STEP 5 → STEP 4 반려)
  → DEV: 결함 수정 + 테스트
  → QA: 재테스트
  → GC: 재검수 (변경 범위에 따라 생략 가능, PM 판단)
```

### 7.4 단계별 검증 매트릭스

각 단계에서 어떤 에이전트가 작업하고 누가 검증하는지 한눈에 표시:

| STEP | 작업자 | 검증자 | 게이트 조건 | 산출물 |
|------|--------|--------|------------|--------|
| 1. 기획서 | GD | PM | 마스터 플랜 일치, 원칙 준수 | 게임 기획서 |
| 2. UX 명세 | UX | GD + PM | 기획 반영, 패턴 준수 | UX 명세서 |
| 3. 엔진 개발 | DEV | QA | 테스트 통과, 성능 충족 | 엔진 코드 + 테스트 |
| 4. UI 개발 | DEV | UX + QA | UX 일치, 기능 정상 | 화면 코드 + 테스트 |
| 5. 통합 QA | QA | PM | 리그레션 PASS, 크로스 게임 정상 | QA 리포트 |
| 6. 챔피언 리뷰 | GC | PM | 재미/밸런스/조작감 PASS | 챔피언 리포트 |
| 7. 빌드/배포 | DEV+PM | - | AAB 생성, 테스트 최종 통과 | AAB + 커밋 |

### 7.5 PM 통제 포인트

PM은 다음 시점에서 **반드시** 개입한다:

| 시점 | PM 행동 |
|------|---------|
| 릴리스 착수 | 마스터 플랜 작업 목록 배포, 에이전트 역할 안내 |
| STEP 1 완료 | 기획서가 마스터 플랜/원칙과 일치하는지 확인 후 STEP 2 진입 승인 |
| STEP 2 완료 | UX 명세가 기획서와 일치하고 아키텍처를 준수하는지 확인 |
| STEP 3→4 전환 | 엔진 테스트 통과 확인 후 UI 개발 승인 |
| STEP 5 완료 | QA 리포트 전체 PASS 확인 후 GC 리뷰 진입 승인 |
| STEP 6 반려 시 | 반려 사유 분석 → 어떤 STEP으로 돌아갈지 결정 |
| STEP 7 완료 | 상태 업데이트, GitHub 커밋 확인, 다음 릴리스 안내 |
| **범위 변경 요청 시** | 마스터 플랜과의 차이 분석 → 사용자 합의 후 플랜 업데이트 |

### 7.6 R1 적용 예시 (Binairo + 게임 허브)

```
PM: "R1 착수합니다. 작업 목록을 배포합니다."
  │
  ├─ STEP 1: GD가 Binairo 기획서 작성
  │   - 규칙: 0/1 채우기, 3연속 금지, 행열 균등, 동일 행열 금지
  │   - 난이도: 6x6(입문) ~ 14x14(마스터)
  │   - 배지 10개, 힌트 4단계, 등급 기준
  │   → PM 검증: "기획서 승인. STEP 2 진행하세요."
  │
  ├─ STEP 2: UX가 화면 흐름 + 인터랙션 설계
  │   - 이진 토글 패턴 적용 (빈칸→0→1→빈칸)
  │   - 게임 허브 카드 레이아웃
  │   - 세로/가로 모드 보드 배치
  │   → GD 검증: "기획 의도 정확히 반영됨"
  │   → PM 검증: "패턴 A 준수, 공유 위젯 활용. STEP 3 진행."
  │
  ├─ STEP 3: DEV가 엔진 구현
  │   - BinairoBoard, Solver, Generator, HintEngine
  │   - 단위 테스트 35개
  │   → QA 검증: "35개 테스트 통과, 100회 생성 전체 유일해, 3초 이내"
  │   → PM: "STEP 4 진행."
  │
  ├─ STEP 4: DEV가 UI 구현
  │   - 플레이/일시정지/결과/튜토리얼 화면
  │   - 게임 허브 리팩토링
  │   → UX 검증: "세로 모드 OK, 가로 모드 셀 크기 조정 필요"
  │   → DEV: 가로 모드 수정
  │   → UX 재검증: "OK"
  │   → QA 검증: "골든 패스 OK, 백키 OK, 백업 OK"
  │   → PM: "STEP 5 진행."
  │
  ├─ STEP 5: QA 통합 테스트
  │   - 스도쿠 640개 + Binairo 35개 = 675개 전체 통과
  │   - 게임 허브 전환 정상
  │   - 백업/복원 라운드트립 OK
  │   → PM: "QA 리포트 전체 PASS. STEP 6 진행."
  │
  ├─ STEP 6: GC 챔피언 리뷰
  │   - 6x6 입문: "1분 만에 완료, 규칙 학습에 적합" ✅
  │   - 10x10 보통: "3연속 금지 추론이 재밌다" ✅
  │   - 14x14 마스터: "15분 소요, 적절한 난이도" ✅
  │   - 힌트: "3단계에서 '이 행은 0이 이미 3개' 설명이 도움됨" ✅
  │   - 조작감: "토글 반응 빠르고 정확" ✅
  │   → GC: "PASS"
  │   → PM: "STEP 7 진행."
  │
  └─ STEP 7: 빌드 & 배포
      - versionCode 4, AAB 빌드
      - Play Store 업로드
      - GitHub 커밋
      → PM: "R1 완료. R2(Minesweeper) 대기 상태입니다."
```
